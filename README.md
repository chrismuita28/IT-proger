# IT-proger

# Главная идея
Реанимация кода в личных (и не только) проектах

# Elevator pitch
У каждого разработчика есть заброшенный проект на GitHub, который жалко удалить, но невозможно восстановить самостоятельно из-за устаревшего кода. Мы создаем C2C-платформу для реанимации легаси-кода, соединяя владельцев с профильными специалистами для обновления и документирования. В отличие от общих бирж фриланса, мы предлагаем узкоспециализированные пакеты услуг именно для работы с чужим кодом. Это решает болезненную проблему для энтузиастов и монетизирует тысячи «мертвых» репозиториев, которые хотят ожить. Конкуренты почти отсутствуют, так как существующие площадки слишком широки для таких точечных задач. В результате заказчик не тратит время и силы на восстановление проекта, а разработчики - дополнительный заработок.

# Функциональные и нефункциональные требования
## Подключение GitHub-репозитория
### 1. ФТ
  * Система должна предоставлять интерфейс авторизации через GitHub
  * Система должна получать список репозиториев пользователя после успешной авторизации и отображать их в виде карточек
  * Система должна позволять пользователю выбрать один репозиторий из предоставленного списка и подтвердить его привязку к создаваемому заказу
  * Система должна проверять доступность репозитория через GitHub API
### 2. НФТ
  * Access- и refresh-токены GitHub не должны храниться в БД в открытом виде. Использовать шифрование
  * Загрузка списка репозиториев и отрисовка интерфейса должны занимать <= 5 секунд
  * Интеграция должна корректно работать с github.com

## Автоматическая оценка сложности и стоимости
## 1. ФТ
  * Система должна автоматически инициировать анализ подключённого репозитория в момент выбора клиентом конкретного пакета услуг
  * Система должна собирать следующие метрики анализируемого кода: количество строк кода (без учёта комментариев и пустых строк), общее число зависимостей, наличие конфигурационных файлов (requirements.txt, Dockerfile и т.д.), данные о покрытии тестами (при наличии)
  * Система должна рассчитывать коэффициент сложности проекта на основе собранных метрик и выбранного пакета услуг
  * Система должна вычислять предварительную стоимость заказа по формуле умножения базовой цены выбранного пакета на коэффициент сложности
  * Система должна отображать результаты оценки в интерфейсе клиента в виде информационной карточки, содержащей детализацию метрик, присвоенный коэффициент сложности, рассчитанную цену

## 2. НФТ
  * Полный сбор метрик и расчёт цены для репозитория должен завершится <= 100 секунд
  * Отклонение автоматической оценки от экспертной не должно превышать 25% в 80% случаев тестовой выборки
  * Сервис оценки должен поддерживать очередь задач и обрабатывать до 100 одновременных запросов

# DDD
## 1. Управление заказами
  * Создает и хранит заказы, связывая клиента, исполнителя и репозиторий
  * Предоставляет выбор готовых пакетов услуг и фиксирует их в заказе
  * Управляет статусами заказа и отслеживает прогресс выполнения
  * Обрабатывает установку, изменение и приостановку дедлайнов
  * Фиксирует факт завершения заказа и инициализирует процесс оплаты

### 2. Анализ репозиториев
  * Авторизует пользователя через GitHub и получает список его репозиториев
  * Сканирует код: собирает LOC, зависимости, наличие тестов и конфигов
  * Рассчитывает коэффициент сложности на основе метрик и выбранного пакета
  * Вычисляет предварительную стоимость

### 3. Пользователи и репутация
  * Регистрирует профили, определяет роли
  * Верифицирует исполнителей: проверяет опыт, портфолио, внешние ссылки
  * Собирает и агрегирует отзывы, пересчитывает динамический рейтинг
  * Предоставляет историю заказов и репутационную статистику по пользователям

### 4. Коммуникация
  * Создаёт приватный чат для каждой пары «клиент–исполнитель»
  * Позволяет исполнителю отправлять промежуточные результаты
  * Реализует механизм апрува/отклонения этапов клиентом
  * Генерирует структурированные отчёты «Что изменено» с диффами и пояснениями

### 5. Платежная система
  * Блокирует средства клиента в эскроу при старте заказа
  * Автоматически переводит оплату исполнителю после подтверждения работы
  * Обрабатывает возвраты и частичные выплаты при отмене/споре

### 6. Арбитраж
  * Принимает заявки на арбитраж, предоставляет модератору историю чата и коммитов
  * Фиксирует решение модератора и обновляет статусы заказа и репутации сторон

# BDD
## Критический путь
Клиент подключает репозиторий -> получает оценку -> создаёт заказ -> исполнитель принимает его

## Успешный сценарий
### Дано:
Клиент авторизован в системе через Github, подключил публичный репозиторий, содержащий файлы программы, а в системе уже зарегистрирован активный прайс-лист с базовой стоимостью пакета услуг.
### Когда:
Клиент выбирает пакет «Обновить зависимости», после чего платформа автоматически запускает фоновый анализ кода, собирает метрики (LOC, число зависимостей, наличие тестов), вычисляет коэффициент сложности, умножает его на базовую цену пакета, отображает итоговую карточку с расчётной стоимостью и сроком, а клиент устанавливает дедлайн и подтверждает создание заказа.
### Тогда:
Заказ фиксируется в статусе pending, мгновенно появляется в общей ленте доступных заданий для верифицированных исполнителей с подходящим стеком, клиент получает системное уведомление об успешном создании, а все собранные метрики, применённая формула расчёта и установленный дедлайн сохраняются в неизменяемой истории заказа.

## Неуспешный сценарий
### Дано:
Клиент авторизован в системе через GitHub и пытается подключить приватный репозиторий, для которого выданный OAuth-токен не имеет права чтения содержимого файлов, а система настроена на обязательное чтение исходного кода для расчёта метрик.
### Когда:
Клиент выбирает пакет «Полный рефакторинг» и инициирует автоматическую оценку, но служба анализа репозиториев при запросе содержимого получает от GitHub API невалидный ответ, что делает невозможным сбор данных о зависимостях, покрытии тестами и реальном объёме кода.
### Тогда:
Процесс оценки прерывается на стороне платформы, заказ не переходит в статус создания, пользовательский интерфейс корректно отрисовывает модальное окно с пояснением об ошибке доступа.
